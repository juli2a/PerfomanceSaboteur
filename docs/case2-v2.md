# Case 2 (v2): Зсув макету через клієнтський стан сайдбару

**Категорія:** Rendering / Візуальна стабільність
**Тумблер:** Rendering → Layout Shift
**Метрика:** CLS

## Короткий опис

Стан згорнутості лівого сайдбару — це персональне налаштування користувача, яке має "пам'ятатись" між заходами. Якщо зберігати його лише в `localStorage`, сервер про нього нічого не знає і завжди рендерить розгорнутий сайдбар; клієнт виправляє це вже після монтування — і через плавну CSS-анімацію ширини це виглядає непомітно для ока, але інструментально це все одно зсув лейауту вже після завантаження сторінки, тобто CLS.

Це замінює попередню версію кейсу 2 (chart без зарезервованої висоти, див. `docs/case2.md`) — обидва варіанти про CLS, а тумблер підтримує лише одну пару "поганого"/"гарного" коду.

---

## Гарний код (Тумблер OFF)

**Реалізація:**
```ts
// app/(shell)/layout.tsx (Server Component)
const cookieStore = await cookies();
const initialCollapsed = cookieStore.get("sidebarCollapsed")?.value === "on";
// ...передається пропом в <Header> і <Sidebar>

// store/sidebar.ts
export const useSidebarStore = create<SidebarState>()((set) => ({
  collapsed: false,
  setCollapsed: (collapsed) => {
    set({ collapsed });
    document.cookie = `sidebarCollapsed=${collapsed ? "on" : "off"}; path=/; max-age=31536000`;
  },
}));

// hooks/useSidebarCollapsed.ts — useSyncExternalStore з initialCollapsed
// як server-снапшот, тому перший клієнтський рендер завжди дорівнює SSR.
```
> Стан дублюється в cookie, яку бачить і сервер (`next/headers`), і клієнт. Обидва незалежно рахують те саме значення з того самого джерела — розбіжності немає, отже й коригувати після монтування нічого.

**Поведінка інтерфейсу:**
- Сайдбар (і зона логотипу в Header) рендериться у правильній ширині одразу, з першого пейнту.
- Перезавантаження сторінки при згорнутому сайдбарі не дає жодного стрибка.
- CLS не зростає від цього елемента.

## Поганий код (Тумблер ON)

**Реалізація:**
```ts
// store/sidebar-unstable.ts
export const useSidebarStoreUnstable = create<SidebarState>()(
  persist(
    (set) => ({ collapsed: false, setCollapsed: (collapsed) => set({ collapsed }) }),
    { name: "sidebar-collapsed-unstable" },
  ),
);
```
> `localStorage` існує лише в браузері. Сервер завжди рендерить `collapsed: false`. `zustand/persist` навмисно не рехідрує стор синхронно на першому рендері (щоб не зловити hydration-warning) — тож і перший клієнтський рендер теж `false`. Одразу після монтування стор оновлюється реальним значенням з `localStorage` — це вже звичайний `setState`, без попереджень React, тому й анімація `transition-[width]` відпрацьовує плавно.

**Поведінка інтерфейсу:**
- Якщо сайдбар був згорнутий раніше — при новому заході він на мить показується розгорнутим, а тоді плавно згортається.
- Зовні виглядає як "просто анімація", але CLS все одно фіксує зсув пікселів, що стались після початкового рендеру.

**Сценарій відтворення (нетиповий для кейсів симулятора — порядок дій має значення):**

Клік по кнопці згортання завжди записує значення в обидва канали одразу (і cookie, і `localStorage`-ключ поганого стору, `hooks/useSidebarCollapsed.ts`) — незалежно від того, який режим зараз активний. Це навмисно: тумблер демонструє лише різницю в МЕХАНІЗМІ читання (cookie vs localStorage-only), а не губить реальний вибір користувача при перемиканні.

1. Згорнути сайдбар стрілкою над іконками навігації (тумблер поки вимкнений — сайдбар збережеться коректно, без стрибка).
2. Увімкнути тумблер "Layout Shift" — клік сам перезавантажує сторінку.
3. На цьому ж перезавантаженні видно короткий стрибок: сайдбар на мить показується розгорнутим, а тоді плавно згортається назад.
4. Для контролю: вимкнути тумблер і перезавантажити сторінку (F5) — стрибка більше немає, сайдбар одразу згорнутий.

Зворотний порядок (спершу тумблер, потім згортання) теж технічно демонструє баг, але вимагає додаткового ручного F5 після згортання, який неочевидний — тому в UI/tip описаний саме порядок вище.

## Аналіз

**Ймовірність успішної демонстрації: 8/10** — ефект реальний і завжди відтворюваний, але потребує одного додаткового кроку користувача (спершу згорнути сайдбар), на відміну від інших кейсів, де досить просто клацнути тумблер.

**Реалізація:** повністю клієнтська різниця в тому, звідки береться персистентність (cookie vs `localStorage`); жодних штучних затримок чи fetch не потрібно — сам механізм гідратації вже дає ефект природно.

**Відповідність UI та кейсу:** ✅ сайдбар — постійний елемент шапки застосунку, є на кожній сторінці.

**Побічний ефект:** оскільки тумблер більше не керує графіком доходів, `SalesChart` стає постійно "гарною" реалізацією (фіксована висота), а Dashboard повертається до природного порядку — банер зверху, графік під ним.

## Мобільна версія

Той самий CLS, але для іншого елемента: замість ширини сайдбару — стан
розгорнутості нижньої панелі метрик (`PerformancePanelMobile`). Механізм
ідентичний десктопній половині кейсу вище (cookie vs `localStorage`-only
hydration mismatch) — тільки застосований до `expanded`, а не до
`collapsed`.

### Зроблено й лишається назавжди (не залежить від тумблера)

Дослідження показало, що app-shell (`body: h-full overflow-hidden`,
`main: overflow-auto`) не давав документу реально скролитись на
мобільному — весь скрол ловив внутрішній `<main>`. Мобільні браузери
ховають адресний рядок лише у відповідь на скрол **документа**, тож без
фіксу кейс фізично не відтворювався на пристрої. Заразом це виявилось і
реальним багом — не працював pull-to-refresh.

- `app/layout.tsx`: `body` — `overflow-hidden` → `overflow-auto lg:overflow-hidden`.
- `app/(shell)/layout.tsx`: `main` втратив `overflow-auto` на мобільному;
  зовнішня обгортка (контейнер `Header`) — `h-full` → `min-h-dvh ... lg:h-full`
  (інакше `sticky`-хедер переставав триматись після одного екрана скролу —
  його "стеля" обмежена висотою контейнера, а не довжиною скролу).
- `app/(shell)/inventory/page.tsx`: сторінка сама тримає
  `h-[calc(100dvh-60px)] lg:h-full` замість успадкованого `h-full` від
  `main` — Inventory (`react-virtual`) потребує означеної висоти-предка,
  якої `main` більше не гарантує на мобільному. Перевірено, що це не
  ламає ні звичайний рендер, ні "плаский" режим без віртуалізації (Case 3
  `heavyMounting`, Case 7 `contextOverhead`) — обидва рендерять всередині
  того самого самодостатнього `scrollRef` (`components/inventory/ProductTable.tsx:208-214`).
- `PerformancePanelMobile` тепер справді `position: fixed` (раніше просто
  тримався внизу, бо шел ніколи не скролився) — `fixed inset-x-0 bottom-0`
  лишається коректним анкерингом незалежно від того, яке джерело
  персистентності зараз керує станом `expanded` (див. нижче).

### Відхилено: `vh`/`dvh` як механізм кейсу

Перша спроба анкерила панель через `h-screen` (статичний `100vh`) замість
`bottom: 0`. **На реальному пристрої підтверджено: CLS від цього не
зростає взагалі.** Причина: браузери навмисно зробили `vh` **статичним**
(дорівнює найбільшій, "схлопнутій" висоті) саме для того, щоб він більше
не перераховувався під час скролу — це й було виправленням старого багу
з "стрибками". Раз рендерений бокс елемента між кадрами не змінюється —
Layout Instability API просто нічого не бачить, CLS рахувати нічого. Баг
реальний (панель ховається під браузерним хромом, поки рядок показаний),
але це occlusion, не layout shift.

Другу ідею — `dvh` у звичайному потоці документа (банер + KPI-рядок на
всю висоту екрана, розтягнутий "вміло" різними одиницями `dvh`/`vh`/`svh`)
— відхилено ще на етапі аналізу, не доходячи до коду: специфікація Layout
Instability API прямо визначає "excluding input" як "будь-яку подію, що
напряму змінює розмір viewport" — тобто саме ховання/поява адресного
рядка. Зсув, який `dvh` дає у відповідь на цю подію, потрапляє у вікно
виключення й не рахується в CLS, скільки не крути. Це той самий підсумок,
що й для `vh` вище — тільки з іншої причини (виключення по вводу, а не
статичність одиниці): якщо проблема відтворюється тільки через скрол —
на CLS вона не вплине.

Обидва варіанти прибрані з коду повністю, без мертвих гілок
(`PanelAnchorUnstable` видалено разом з файлом `PanelAnchor.tsx`, стилі
анкерингу перенесені прямо в `PerformancePanelMobile.tsx`) — код-конвенція
репозиторію не лишає "residue" від відхилених варіантів; історія
лишається тільки тут, у доці.

### Нюанс, якого немає в десктопній половині: `PerformancePanel` ніколи не рендериться на сервері

`PerformancePanel` визначає mobile/desktop через `isMobile` з
`MediaContext` (`useMediaQuery` — чистий клієнтський `matchMedia`, без
UA-визначення на сервері) і повертає `null`, поки `isMobile === undefined`
(`components/simulator/performance-panel/PerformancePanel.tsx`). Це
означає: на відміну від `Header`/`Sidebar`, які завжди присутні в
реальному SSR HTML (і просто ховаються CSS-класами на потрібних
брейкпоінтах), **весь `PerformancePanel` — обидва варіанти — відсутній у
сирому SSR HTML і монтується вже суто на клієнті**, вже після того як
`MediaContext` визначиться. Підтверджено `curl`'ом: `sim-panel-mobile` не
з'являється в жодній відповіді сервера, з жодною комбінацією cookie.

Наслідок для механізму: сайдбарів трюк `useSyncExternalStore` +
`getServerSnapshot` працює лише під час справжньої гідратації існуючого
SSR DOM. Оскільки для цього піддерева такого DOM ніколи не існує, React
монтує його як звичайний клієнтський mount — і одразу читає поточне
значення стору (`false` за замовчуванням), а не `initialExpanded`.
Перша спроба виправити це локально (сідити стор синхронно під час
першого рендера, без SSR) прибирала стрибок для "гарного" шляху — але й
для "поганого" теж: на живому пристрої панель з'являлась одразу в
правильному положенні, без жодного стрибка. Причина — `PerformancePanelMobile`
монтується не одразу (чекає на `isMobile`), а рехідрація
`store/panel-expanded-unstable.ts` з `localStorage` встигає завершитись
до цього моменту, тож перший рендер уже бачить правильне значення.
Глухий кут того самого класу, що й `vh`/`dvh` вище — тільки причина інша.

**Фінальне рішення:** ще одна cookie, `isMobile`, яка дзеркалить
`MediaContext`'s значення (пишеться в `context/MediaContext.tsx`, у тому
самому місці, де `isMobile` обчислюється — один ефект на весь застосунок).
`app/(shell)/layout.tsx` читає її на сервері й передає пропом
(`initialIsMobile`) у `PerformancePanel`, де вона працює як fallback:
```ts
const isMobile = useContext(MediaContext) ?? initialIsMobile;
```
Це не зміна `MediaContext`/`useMediaQuery` — вони лишаються без змін,
усі інші споживачі `isMobile` (`ProductTable.tsx` — Case 3
`heavyMounting`, Case 7 `contextOverhead`; `CategoryFilter.tsx`) далі
чекають на `matchMedia` як і раніше. Зміна лише в `PerformancePanel`:
коли cookie `isMobile` вже є (будь-який повторний візит), компонент
рендерить правильну гілку одразу на сервері — і **справді бере участь у
гідратації**, а не монтується "з нуля" на клієнті. Це відновлює точну
відповідність сайдбарному механізму: `useStableExpanded` повернуто на
`useSyncExternalStore` + ефект-реседінг (як у `hooks/useSidebarCollapsed.ts`),
а не синхронний seed під час рендера — тепер, коли `PerformancePanelMobile`
дійсно рендериться на сервері, такий синхронний запис мутував би спільний
модульний Zustand-стор між паралельними запитами.

### Гарний код (Тумблер OFF)

**Реалізація:**
```ts
// app/(shell)/layout.tsx (Server Component)
const initialExpanded = isLayoutShiftOn
  ? false
  : cookieStore.get("panelExpanded")?.value === "on";
// ...передається пропом у <PerformancePanel> → <PerformancePanelMobile>

// store/panel-expanded.ts
export const usePanelExpandedStore = create<PanelExpandedState>()((set) => ({
  expanded: false,
  setExpanded: (expanded) => set({ expanded }),
}));

// hooks/usePanelExpanded.ts — useSyncExternalStore з initialExpanded
// як server-снапшот, тому перший клієнтський рендер завжди дорівнює SSR
// (працює лише завдяки cookie isMobile вище — див. підрозділ над цим).
function useStableExpanded(initialExpanded: boolean) {
  const expanded = useSyncExternalStore(
    usePanelExpandedStore.subscribe,
    () => usePanelExpandedStore.getState().expanded,
    () => initialExpanded,
  );
  useEffect(() => {
    usePanelExpandedStore.setState({ expanded: initialExpanded });
  }, [initialExpanded]);
  return expanded;
}
```
> Той самий принцип, що й у сайдбара вище: стан дублюється в cookie, яку
> бачить і сервер, і клієнт — розбіжності немає, коригувати після
> монтування нічого.

**Поведінка інтерфейсу:**
- Панель рендериться у правильній висоті (50px згорнута / 174px
  розгорнута) одразу, з першого пейнту.
- Перезавантаження сторінки при розгорнутій панелі не дає жодного стрибка.
- CLS не зростає від цього елемента.

### Поганий код (Тумблер ON)

**Реалізація:**
```ts
// store/panel-expanded-unstable.ts
export const usePanelExpandedStoreUnstable = create<PanelExpandedState>()(
  persist(
    (set) => ({ expanded: false, setExpanded: (expanded) => set({ expanded }) }),
    { name: "panel-expanded-unstable" },
  ),
);
```
> `localStorage` існує лише в браузері. Сервер завжди рендерить
> `expanded: false`. `zustand/persist` не рехідрує стор синхронно на
> першому рендері — тож перший клієнтський рендер теж `false`. Одразу
> після монтування стор оновлюється реальним значенням з `localStorage` —
> звичайний `setState`, без попереджень React. Панель зафіксована знизу
> (`fixed; bottom: 0`), тож зростання висоти (50px → 174px) фізично
> зсуває її верхній край вгору між двома кадрами — прямий, безумовний
> зсув rect реального елемента, ніяк не пов'язаний зі скролом чи resize
> viewport (на відміну від відхиленого `dvh`-варіанту вище), тому жодних
> винятків CLS тут немає.

**Поведінка інтерфейсу:**
- Якщо панель була розгорнута раніше — при новому заході вона на мить
  показується згорнутою (50px), а тоді розгортається (174px).
- Підтверджено на живому пристрої: стабільний приріст CLS **+0.07** на
  кожному такому перезавантаженні — узгоджується з прорахованим
  діапазоном (≈0.03–0.06 на типовому мобільному viewport). У жовту зону
  ("Needs improvement", >0.1) сам по собі не виводить, але приріст видимий
  і стабільно відтворюваний — для демонстрації цього достатньо.

**Сценарій відтворення (той самий порядок дій, що й для сайдбару):**

Клік по кнопці розгортання завжди записує значення в обидва канали одразу
(і cookie, і `localStorage`-ключ поганого стору) — незалежно від того,
який режим зараз активний.

1. Розгорнути панель кнопкою (тумблер поки вимкнений — стан збережеться
   коректно, без стрибка).
2. Увімкнути тумблер "Layout Shift" — клік сам перезавантажує сторінку.
3. На цьому ж перезавантаженні видно короткий стрибок: панель на мить
   показується згорнутою, а тоді розгортається назад.
4. Для контролю: вимкнути тумблер і перезавантажити сторінку (F5) —
   стрибка більше немає, панель одразу в правильному стані.

Зворотний порядок (спершу тумблер, потім розгортання) теж технічно
демонструє баг, але вимагає додаткового ручного F5 після розгортання —
тому в UI/tip описаний саме порядок вище.

### Прод/локальна розбіжність (2026-07-15) і фікс

На Vercel-проді той самий репро віддавав CLS ≈0 в більшості спроб (рух
панелі все одно було видно на власні очі), тоді як локально (`next dev` і
навіть прод-білд через `next start`) зсув фіксувався стабільно щоразу. При
одночасно увімкненому Case 5 (Waterfall) CLS на проді піднімався **завжди**
— цей збіг і вказав на справжню причину.

`zustand`'s `persist` рехідрується з `localStorage` синхронно, під час
створення стору (до першого клієнтського рендеру) — не пізнішим `setState`
після монтування, як раніше стверджував коментар у
`store/panel-expanded-unstable.ts`. Видиму "пізнішу" корекцію насправді
забезпечував сам `useSyncExternalStore`: перший клієнтський прохід рендерить
`getServerSnapshot()`, щоб збігтись із SSR, і лише після монтування
підхоплює вже інше живе значення стору — через внутрішній layout-effect,
тобто до пейнту. Єдине, що визначало, чи побачить користувач (і
Layout Instability API) "неправильний" кадр — чи встигав браузер відмалювати
сирий SSR HTML до того, як довантажиться й виконається JS-бандл гідратації.
На Vercel (CDN, кешовані статичні чанки) цей бандл зазвичай встигав раніше —
окремого "до"-кадру не було. З увімкненим Waterfall банер/KPI і панель
гарантовано приходять в одному заблокованому SSR-response через кілька
секунд — тож "до"-кадр встигав відмалюватись завжди, без винятків.

**Фікс** (`hooks/usePanelExpanded.ts`): `useUnstableExpanded` тепер сама
відображає `initialExpanded`, а на реальне значення з `localStorage`
перемикається лише через `setTimeout(..., 150)`. В історії кейсу це не
окремий свідомий "хак заради демо" — той самий розробник, що писав "поганий"
шлях, помітив, що анімація відновлення стану не програється (виправлене
значення прилітало до того, як браузер узагалі домалював початковий стан —
transition-у не було звідки стартувати), і заглушив це вгаданою затримкою
замість того, щоб розібратись у причині. 150мс — з великим запасом над
будь-якою реальною різницею в швидкості гідратації, тому саме ця недбалість
робить зсув стабільно відтворюваним незалежно від платформи.
