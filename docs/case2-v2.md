# Case 2 (v2): Зсув макету через клієнтський стан сайдбару

**Категорія:** Rendering / Візуальна стабільність
**Тумблер:** Rendering → Layout Shift
**Метрика:** CLS

## Короткий опис

Стан згорнутості лівого сайдбару — це персональне налаштування користувача, яке має "пам'ятатись" між заходами. Якщо зберігати його лише в `localStorage`, сервер про нього нічого не знає і завжди рендерить розгорнутий сайдбар; клієнт виправляє це вже після монтування — і через плавну CSS-анімацію ширини це виглядає непомітно для ока, але інструментально це все одно зсув лейауту вже після завантаження сторінки, тобто CLS.

Це замінює попередню версію кейсу 2 (chart без зарезервованої висоти, див. `docs/case2.md`) — обидва варіанти про CLS, а тумблер підтримує лише одну пару "поганого"/"гарного" коду.

---

## Гарний код (Тумблер OFF)

**Реалізація:**
```ts
// app/(shell)/layout.tsx (Server Component)
const cookieStore = await cookies();
const initialCollapsed = cookieStore.get("sidebarCollapsed")?.value === "on";
// ...передається пропом в <Header> і <Sidebar>

// store/sidebar.ts
export const useSidebarStore = create<SidebarState>()((set) => ({
  collapsed: false,
  setCollapsed: (collapsed) => {
    set({ collapsed });
    document.cookie = `sidebarCollapsed=${collapsed ? "on" : "off"}; path=/; max-age=31536000`;
  },
}));

// hooks/useSidebarCollapsed.ts — useSyncExternalStore з initialCollapsed
// як server-снапшот, тому перший клієнтський рендер завжди дорівнює SSR.
```
> Стан дублюється в cookie, яку бачить і сервер (`next/headers`), і клієнт. Обидва незалежно рахують те саме значення з того самого джерела — розбіжності немає, отже й коригувати після монтування нічого.

**Поведінка інтерфейсу:**
- Сайдбар (і зона логотипу в Header) рендериться у правильній ширині одразу, з першого пейнту.
- Перезавантаження сторінки при згорнутому сайдбарі не дає жодного стрибка.
- CLS не зростає від цього елемента.

## Поганий код (Тумблер ON)

**Реалізація:**
```ts
// store/sidebar-unstable.ts
export const useSidebarStoreUnstable = create<SidebarState>()(
  persist(
    (set) => ({ collapsed: false, setCollapsed: (collapsed) => set({ collapsed }) }),
    { name: "sidebar-collapsed-unstable" },
  ),
);
```
> `localStorage` існує лише в браузері. Сервер завжди рендерить `collapsed: false`. `zustand/persist` навмисно не рехідрує стор синхронно на першому рендері (щоб не зловити hydration-warning) — тож і перший клієнтський рендер теж `false`. Одразу після монтування стор оновлюється реальним значенням з `localStorage` — це вже звичайний `setState`, без попереджень React, тому й анімація `transition-[width]` відпрацьовує плавно.

**Поведінка інтерфейсу:**
- Якщо сайдбар був згорнутий раніше — при новому заході він на мить показується розгорнутим, а тоді плавно згортається.
- Зовні виглядає як "просто анімація", але CLS все одно фіксує зсув пікселів, що стались після початкового рендеру.

**Сценарій відтворення (нетиповий для кейсів симулятора — порядок дій має значення):**

Клік по кнопці згортання завжди записує значення в обидва канали одразу (і cookie, і `localStorage`-ключ поганого стору, `hooks/useSidebarCollapsed.ts`) — незалежно від того, який режим зараз активний. Це навмисно: тумблер демонструє лише різницю в МЕХАНІЗМІ читання (cookie vs localStorage-only), а не губить реальний вибір користувача при перемиканні.

1. Згорнути сайдбар стрілкою над іконками навігації (тумблер поки вимкнений — сайдбар збережеться коректно, без стрибка).
2. Увімкнути тумблер "Layout Shift" — клік сам перезавантажує сторінку.
3. На цьому ж перезавантаженні видно короткий стрибок: сайдбар на мить показується розгорнутим, а тоді плавно згортається назад.
4. Для контролю: вимкнути тумблер і перезавантажити сторінку (F5) — стрибка більше немає, сайдбар одразу згорнутий.

Зворотний порядок (спершу тумблер, потім згортання) теж технічно демонструє баг, але вимагає додаткового ручного F5 після згортання, який неочевидний — тому в UI/tip описаний саме порядок вище.

## Аналіз

**Ймовірність успішної демонстрації: 8/10** — ефект реальний і завжди відтворюваний, але потребує одного додаткового кроку користувача (спершу згорнути сайдбар), на відміну від інших кейсів, де досить просто клацнути тумблер.

**Реалізація:** повністю клієнтська різниця в тому, звідки береться персистентність (cookie vs `localStorage`); жодних штучних затримок чи fetch не потрібно — сам механізм гідратації вже дає ефект природно.

**Відповідність UI та кейсу:** ✅ сайдбар — постійний елемент шапки застосунку, є на кожній сторінці.

**Побічний ефект:** оскільки тумблер більше не керує графіком доходів, `SalesChart` стає постійно "гарною" реалізацією (фіксована висота), а Dashboard повертається до природного порядку — банер зверху, графік під ним.
