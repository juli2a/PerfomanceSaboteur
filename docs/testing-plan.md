# План: тестове покриття PerfSaboteur

## Контекст

Продукт готовий, документація (`docs/*.md`, tip-об'єкти в `lib/simulator-cases.ts`) вже приведена у відповідність до коду. Тестів у проєкті зараз немає взагалі — жодного тестового фреймворку в `package.json`, жодного `*.test.ts` файлу. Мета: закрити тестами не все підряд, а місця з реальною, нетривіальною логікою — і зробити це так, щоб тести перевіряли **задокументовану поведінку**, а не випадково зафіксували баг, який уже сидить у коді. Другорядна, але важлива мета — цей процес має бути навчальним: я пояснюю кожен крок, а не просто видаю файли.

## Частина 1 — як не "затестити" баг

Проблема чесна: якщо дивитись у реалізацію і одразу писати `expect(fn(x)).toBe(<що функція зараз повертає>)`, тест стає дзеркалом коду, а не перевіркою коректності. Він завжди зелений і не ловить нічого. Підхід, який цьому протидіє:

1. **Спочатку — незалежне джерело істини, а не implementation.** У цьому проєкті воно вже є, і це рідкість для pet-проєкту:
   - `docs/data.md` — точні формули (`LTV = id*1250 + age*300`, `avgCheck = revenue/orders`, правила `logisticStatus`, формула `deriveRealProductId`);
   - `docs/case1.md … case8.md` + поля `problem/effect/badCode/goodCode/summary` у `lib/simulator-cases.ts` — словесний опис того, що саме має статись у "поганому" і "хорошому" шляху кожного антипатерну.
   - Це написано декларативно, до того як я дивлюсь у тіло функції. Тому я формулюю очікування **з документа**, а не з коду.

2. **Порядок дій для кожного тесту:**
   a. Читаю опис у docs / tip → своїми словами формулюю, яка поведінка очікується;
   b. Пишу assertion на основі цього опису;
   c. Тільки тепер відкриваю реалізацію — і звіряю. Якщо код збігається — добре, тест лишається. Якщо код **не** збігається з описом — я не підганяю тест під код мовчки. Натомість виношу це окремим пунктом: "розбіжність зі спекою в X" — і ми вирішуємо разом, це баг чи документація застаріла.
   d. Для суто структурних речей (назва експорту, сигнатура функції) звірка з кодом — це нормально, це не "логіка", а контракт, з яким тест все одно мусить узгоджуватись.

3. **Де docs мовчать, а логіка тонка** (наприклад, `merge`/`partialize` у `store/simulator-control.ts`) — джерело істини тоді коментарі-"чому" в самому коді (вони пояснюють намір, а не просто описують, що робить рядок). Тест пишеться на **інваріант**, сформульований з цього наміру ("cookie-кейси завжди беруться з cookie-seeded стану, а не з localStorage-блоба"), а не на конкретне число, яке код зараз повертає.

4. Кожен новий test suite я одразу запускаю і показую вам реальний результат (зелений/червоний), а не просто кажу "готово".

5. **Цей документ — мапа, не покейсовий список.** Частина 3 нижче описує *що* і *чому* цікаво тестувати по кожній функції/хуку/компоненту, але не перелічує заздалегідь кожен конкретний `it(...)`. Точний список тест-кейсів (з конкретними вхід/вихід значеннями) я формулюю безпосередньо перед тим, як писати тести для конкретного файлу — показую його вам як короткий чекліст, і тільки після цього пишу код.

## Частина 2 — які види тестів і навіщо кожен

Канонічна трійка — **Unit / Integration / E2E**. Критерій поділу не "яка тека чи модуль", а скільки реальних (немокнутих) колаборантів залучено:

- **Unit** — одна ізольована одиниця логіки. Без реального React, без мережі, без браузерних API.
- **Integration** — кілька реальних частин працюють РАЗОМ, і тест перевіряє саме стик між ними.
- **E2E** — через справжній застосунок повністю, як користувач.

За сучасним каноном фронтенд-тестування (Testing Trophy, Kent C. Dodds) найбільше довіри дає саме Integration-шар — "чим більше тест нагадує реальне використання, тим більше довіри він дає" — тому в цьому проєкті він не другорядний, а основний.

| Рівень | Інструмент | Що перевіряє | Чому саме тут |
|---|---|---|---|
| **Unit** | Vitest | чисті функції (`derive.ts` тощо) **+** увесь шар Zustand-сторів, включно з `merge`/`partialize` | найдешевші, найстабільніші, ловлять помилки у формулах і в логіці стану |
| **Integration — Hooks** | Vitest + `@testing-library/react` (`renderHook`) | кастомні хуки: `renderHook` монтує справжній React і хуки координують React + store + browser API (cookie, location) або мережу — стик кількох реальних частин | тут живе механіка race condition, SSR-cookie sync |
| **Integration — Component** | Vitest + RTL + `msw` | взаємодія компонент ⇄ store ⇄ мережа, товщий зріз того самого шару | перевірка "кейс увімкнено → видно правильний ефект" |
| **E2E** | Playwright | реальний браузер, реальний Next.js dev/build сервер | єдиний спосіб чесно перевірити SSR-кейси (cookie → reload → інший HTML) і те, що бачить живий користувач |

**Store — завжди Unit, без винятків.** Zustand-стор сам по собі — ізольований об'єкт зі станом; `getState()/setState()` не зачіпають ні React, ні мережу, ні DOM. Це стосується і `merge`/`partialize` у `simulator-control.ts`: не потрібно ні рефакторити код, ні проганяти стор крізь реальний localStorage — Zustand-мідлвар `persist` сам виставляє ці функції через `store.persist.getOptions()`. У тесті викликаємо `useSimControlStore.persist.getOptions().merge(persistedBlob, currentState)` напряму, як звичайну чисту функцію — жодного дотику до localStorage, тож це лишається Unit.

**Свідомо поза межами:** точні числа CWV (LCP/CLS/INP) в e2e — вони нестабільні в CI й гілку логіки не перевіряють; presentational-обгортки в `components/ui/`; вміст `design/` (це макети, не app-код); візуальний regression-тестінг (screenshot diffing) — окрема тема, не зараз.

## Частина 3 — конкретні кандидати (не 100%, тільки цікаве)

### Unit — чисті функції
- **`lib/utils/derive.ts`**: `deriveTrend` (edge case: усі значення рівні → `false`, відкат до попереднього сегмента, якщо останній "плаский"); `deriveLtv` (формула з `docs/data.md`, а не з коду); `deriveHalfWindowDeltaPercent` (гілка `older === 0`); `deriveKpiTrend` (задокументована гарантія "завжди ≥2%", попри ±4% wobble — саме той випадок, де рандомний шум міг би зламати гарантію, якщо мінімум прибрати).
- **`lib/server/inventory.ts`**: `deriveRealProductId` як точна інверсія формули ампліфікації id (`((id-1) % 100) + 1`) — round-trip тест для всіх 20 батчів; правила `logisticStatus` (stock ≤3 / ≤10 / інакше — по shipping) — таблично з `docs/data.md`.
- **`lib/server/dashboard.ts`**: сортування "топ-5 по LTV" (а не перші 5 за порядком API), агрегація revenue/avgCheck.
- Вибірково `lib/utils/gauge.ts`, `sparkline-processing.ts` — там, де є розгалуження, а не тривіальний форматер.

### Unit — Zustand-стори
- **`store/simulator-control.ts`** — найцікавіше місце в проєкті: `partialize` виключає `SSR_COOKIE_CASES` з localStorage-блоба, `merge` бере ці ключі з cookie-seeded стану, а не з persisted. Тест викликає `useSimControlStore.persist.getOptions().merge(persistedBlob, currentState)` напряму (без реального localStorage): якщо persisted blob містить `waterfall: true`, а поточний (cookie-seeded) стан — `waterfall: false`, результат merge має лишити `false` (з cookie), а не `true` (з localStorage) — і навпаки для НЕ-cookie кейсів. Це саме той тип логіки, де легко помилитись на рефакторингу і де баг не видно неозброєним оком.
- `triggerAlert`/`dismissAlert`/`closeAlert` — інваріант ідемпотентності (`triggerAlert` двічі не змінює вже `"shown"`), `closeAlert` видаляє ключ повністю, а не ставить `"dismissed"`.
- **`store/render-counter.ts`**: `startTracking` завжди скидає лічильник, `startTrackingIfIdle` скидає лише якщо не трекає (дозволяє накопичувати серію дій в один "сплеск"), `increment` ігнорується, якщо `!isTracking`.

### Integration — Hooks
- **`hooks/useInventorySearch.ts`** — серце Case 4 (race condition). Через `msw` + `vi.useFakeTimers()`: bad path — відповідь, що прийшла останньою за часом, перезаписує стан, навіть якщо належить старішому запиту (відтворюємо приклад із `docs/case4.md`: короткий запит відповідає повільніше й усе одно "перемагає"); good path — `AbortController` гарантує, що застаріла відповідь ніколи не долітає до `setMatchedIds`.
- **`hooks/useToggleCase.ts`** — для `SSR_COOKIE_CASES` виставляє `document.cookie` і викликає `window.location.reload()`, для інших кейсів — ні.
- **`hooks/useSyncSsrCookies.ts`** — синхронізує store з cookie **тільки** якщо значення розійшлись; тест на "не викликає `setToggle` зайвий раз, якщо вже синхронізовано" (зайвий виклик = зайвий ре-рендер, а весь продукт про це).
- **`hooks/useClearAlertsOnNavigate.ts`** — очищає алерти при зміні `pathname`, але **не** на самому першому запуску ефекту. Причина цього винятку задокументована прямо в коментарі коду: SSR-кейс (наприклад, Case 6, Hydration Mismatch) може викликати свій алерт синхронно під час гідратації цієї ж сторінки, ще до того, як цей ефект взагалі встигне відпрацювати вперше — якби очищення відбувалось безумовно на кожному запуску, воно стерло б щойно показаний алерт замість того, щоб дати йому лишитись видимим. Тест: рендеримо хук з якимсь `pathname`, встановлюємо алерт напряму через store, змінюємо `pathname` (через ререндер з новим значенням від `usePathname`) — алерт має зникнути тільки на цій, другій, зміні, а не одразу після першого монтування хука.
- **`hooks/useResetAllToggles.ts`** — задокументована в коментарі гарантія: "All off" викликає `window.location.reload()` щонайбільше один раз за весь виклик, а не по одному разу на кожен активний `SSR_COOKIE_CASES`-кейс. Тест: вмикаємо кілька SSR-кейсів одночасно, викликаємо повернуту функцію, рахуємо кількість викликів мокнутого `window.location.reload` — має бути рівно один (а не нуль і не кілька), і рівно нуль, якщо жоден активний кейс не належав до `SSR_COOKIE_CASES`.
- **`hooks/useRerenderNodesReporter.ts`** — таймер на 100мс (`SETTLE_DELAY_MS`), який перезапускається на кожен новий `increment` у `render-counter` store, поки триває "сплеск" рендерів; тільки коли серія затихла — публікує підсумковий рахунок у `simulator-performance` store і вирішує, викликати `triggerAlert` чи `closeAlert`, залежно від того, чи рахунок перевищив `alertThreshold`. Через `renderHook` + `vi.useFakeTimers()`: перевіряємо, що кілька `increment` поспіль **не** публікують проміжні значення (таймер щоразу скидається), і що межове значення рахунку (рівно `alertThreshold` проти `alertThreshold + 1`) веде до різних гілок (`closeAlert` проти `triggerAlert`).
- **`hooks/useSyncControlsAcrossBreakpoint.ts`** — при переході `isMobile` з `false` на `true` примусово відкриває мобільну панель кейсів, але тільки якщо є активний `activeGuideKey`; при переході назад на десктоп — завжди закриває. Захищено від зайвого спрацювання через `wasMobileRef` (реагує лише на реальну зміну breakpoint-а, а не на кожен рендер) і від невизначеного `isMobile` (SSR-стан "ще не знаємо"). Тест: змінюємо значення, яке повертає замоканий `useContext(MediaContext)`, між рендерами хука і перевіряємо виклики (чи невиклики) `setControlsOpen` для кожної з трьох гілок — mobile-з-гідом, mobile-без-гіда, назад-на-десктоп.
- **`hooks/usePanelExpanded.ts`** і **`hooks/useSidebarCollapsed.ts`** — механіка Case 2 (layout shift) на hook-рівні, а не тільки на E2E. Обидва хуки на кожен клік пишуть **в обидва** стори одночасно (стабільний cookie-backed і нестабільний localStorage-only) незалежно від того, який зі сторів зараз керує відображенням — задокументований намір: перемикання Case 2 toggle не повинно губити поточний стан користувача. У `usePanelExpanded` додатково є штучна затримка `setTimeout(..., 150)` на "поганому" шляху — відтворення реального бага (хтось силою придушив візуальний глюк здогадкою про затримку замість того, щоб знайти причину). Дотепер єдине покриття Case 2 в усьому плані — один E2E-сценарій, що перевіряє лише початковий SSR HTML (Частина 3, E2E, нижче); жоден рівень тестів не перевіряв ні подвійний запис, ні поведінку хука вже після гідратації. Тест: клік по `setExpanded`/`setCollapsed` — перевіряємо, що обидва стори (через `getState()`) отримали нове значення і що cookie виставлена; окремо для `usePanelExpanded` — що значення, яке хук повертає в "нестабільному" режимі, з'являється лише після проходження таймера (`vi.advanceTimersByTime`), а не одразу.

### Integration — Component
- **Case 7 (Context storm) vs Zustand-selector**: рендеримо рядки таблиці, клікаємо один чекбокс, через `render-counter` перевіряємо — при вимкненому кейсі ре-рендерився тільки один рядок, при увімкненому (симуляція через Context) — усі видимі.
- **Case 8 (broken memoization)**: аналогічно — тільки одна картка ре-рендериться при зміні відповідного продукту, коли toggle вимкнено.
- **ControlPanelTogglers**: клік по тоглу → `store.toggles` оновлюється → для `SSR_COOKIE_CASES` викликається (мокнутий) reload, для решти — ні.

### E2E (Playwright)
Буквально відтворюємо кроки "Reproduction" з `docs/caseN.md`:
- Case 5 (waterfall): toggle on → cookie виставлено → reload → мережеві запити йдуть послідовно (timing через `page.route`) vs паралельно при off.
- Case 2 (layout shift): toggle + cookie → перший SSR HTML містить очікуваний початковий стан сайдбару.
- Case 6 (hydration mismatch): консольна помилка гідратації з'являється тільки при toggle on.
- Case 4 (race condition): реальний ввід у полі пошуку — таблиця зрештою показує правильний результат; проміжний неправильний стан спостережуваний тільки при увімкненому кейсі.

## Частина 4 — інструментарій, який треба додати

Нічого з цього зараз не стоїть у `package.json`:
- `vitest` + `@vitejs/plugin-react` + `jsdom` — unit/store/hook/component рівень; обрано замість Jest, бо ESM-native і швидший на Vite-подібній конфігурації, добре дружить з Next.js 16 App Router.
- `@testing-library/react` + `@testing-library/jest-dom` + `@testing-library/user-event` — компонентні тести.
- `msw` — перехоплення `fetch` до DummyJSON у integration-тестах: без нього тести або б'ють по реальному API (нестабільно, повільно), або кожен тест вручну мокає `global.fetch` (крихко).
- `@playwright/test` — e2e, реальний браузер.
- Скрипти в `package.json`: `test`, `test:watch`, `test:e2e`, `test:coverage` (coverage-звіт як орієнтир, не як поріг у 100%).

## Частина 5 — порядок виконання

Рухаємось шматками, кожен — окремий крок з поясненням і реальним запуском тестів перед тим, як переходити далі:

1. **Setup**: `vitest` + RTL + `msw`, базовий конфіг (аліас `@/*`, jsdom-середовище, setup-файл для `jest-dom`). Playwright поки не чіпаємо — його конфіг окремий і не потрібен для перших кроків.
2. **Unit — чисті функції**: `lib/utils/derive.ts` + `lib/server` (`logisticStatus`, `deriveRealProductId`, top-5-LTV) — найбезпечніше місце для старту, показую сам патерн "спека → тест → звірка з кодом".
3. **Unit — Zustand-стори**: `simulator-control.ts` (merge/partialize через `persist.getOptions()`) + `render-counter.ts` — тут детально розберу, як тестувати Zustand-стор без рендеру React-дерева і без реального localStorage.
4. **Integration — Hooks**: `useSyncSsrCookies`, `useToggleCase`, `useInventorySearch` (з `msw` + `vi.useFakeTimers()`), `useClearAlertsOnNavigate`, `useResetAllToggles`, `useRerenderNodesReporter`, `useSyncControlsAcrossBreakpoint`, `usePanelExpanded`, `useSidebarCollapsed` — найскладніший з двох Integration-рівнів, бо є час і побічні ефекти.
5. **Integration — Component**: Case 7 / Case 8 тести на кількість ре-рендерів.
6. **E2E**: окремий Playwright-setup + 3–5 сценаріїв для мережевих/SSR-кейсів (5, 2, 6, 4), де Unit/Integration недостатньо для чесної перевірки.

Після кожного кроку — короткий підсумок: що покрито, чому саме так, які trade-off'и (наприклад, чому `msw`, а не реальний DummyJSON; чому `renderHook`, а не рендер повного дерева).

## Верифікація
- Кожен suite реально запускається (`pnpm test`) і показує справжній зелений/червоний статус.
